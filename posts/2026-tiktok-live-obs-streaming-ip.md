# 一、TikTok 美区直播的真实网络挑战

## 1.1 TikTok 推流的链路特征

- RTMP 推流走 TCP，单向，低延迟。
- 推流端必须与 TikTok 推流服务器（CDN）建立长时间连接，可能从开始到结束一直挂在那里（最长通常 4-8 小时）。
- 推流期间敏感参数：丢包率（< 1%）、抖动（< 30ms）、RTT（< 200ms 且稳定）、上行带宽（≥ 6Mbps 1080p）。

## 1.2 TikTok 端的风控信号

- 推流期间 IP 段稳定性（中途不变 IP、不变 ISP）。
- IP 信誉（不能是机房 IP / 高风险列表）。
- 推流端地理位置（美区直播必须美区 IP，不能是中国 IP 假冒地理）。
- 账号绑定的直播网络环境（同一账号多 IP 切换会被审）。

## 1.3 OBS 推流的坑

- OBS 默认直连推流服务器。团队通常会装 NDI/SocksCap，让 OBS 走不同的本地代理。
- OBS 的 RTMP 实时性敏感，延迟过高会影响画面同步。

# 二、市面"IP 推流包"的坑盘点

2025-2026 常见三类翻车：

| 坑 | 说明 | 后果 |
|-----|------|------|
| 把"机房 IP 标住宅"卖给推流团队 | WHOIS 是 BGP 路由，但 scamscore 显示 datacenter | 推流期间 5-30 分钟后被 TikTok 限流或下播 |
| 动态住宅 IP 用作 RTMP 长连接 | IP 每 1-30 分钟换一次 | OBS 重连失败、TikTok 检测异常/下播 |
| 住宅 IP 轮换协议错误 | SOCKS5 half-proxy 不稳定 | RTMP 断流、画面卡顿、互动延迟 |

# 三、推荐方案：静态住宅 IP + OBS 系统代理

2026 年跑美区直播最稳的姿势：

1. **一台固定美区城市的静态住宅 IP 的 Windows 推流机器（OBS 主机）**，挂在 NDI/SocksCap64 代理上，全系统走这个 IP 出口。
2. **账号端操作（用另一台/手机端）**：账号管理、商品上架、直播预告发布，都走这台美区静态住宅 IP。账号和推流的 IP 在同段，TikTok 端认为是"同一个人/同网络"在直播。
3. **互动/流量端**：主播出镜用手机、互动监控用电脑、评论管理用手机，跟推流机器解耦（避免 ip 过多 → 风控）。

# 四、OBS 端具体配置

## 4.1 系统代理（SocksCap64 + SOCKS5）

- 在推流机器上装 SocksCap64，把 OBS.exe / obs64.exe 拖进去，强制走 SOCKS5。
- 代理指向供应商给的住宅 IP 节点。节点要有"长连接稳定"标签。

## 4.2 直播伴侣（TikTok Live Creator 桌面版）

- 如果用官方桌面推流工具"直播伴侣"，同样走 SocksCap64 代理化。
- 绑定时填 RTMP URL：默认 TikTok 美区默认在 rtmplive.tiktok.com / 美国节点。

## 4.3 NDI 链路（如果用多机位）

- 主推流机收 NDI 信号，副机只发 NDI 局域网流。NDI 优先级高，但同样的副机也要走静态住宅 IP。

# 五、压力与稳定性自测

开直播前，做这些预演：

- 用 OBS 录 1 小时本地压片，确认上行速率稳定不掉帧。
- 用 Speedtest 服务器测到 TikTok 推流服务器的 RTT（应 < 200ms）。
- 丢包测试：连续 ping 推流主机 1000 包，丢包率 < 0.5%。
- 跨 ISP 验证：试试本地 ISP、邻居家的 Wi-Fi、4G 切换，确认上游网络本身没问题。

# 七、常见误区

- **"换 IP 频繁防止关联"？**：美区直播账号是固定 IP 模式，频繁换会触发风控，反而被限流。
- **"住宅 IP 一定比机房 IP 快"？**：不一定。但住宅 IP 的"低 RTT + 高信誉"更稳定。
- **"多账号同 IP 推流"？**：绝对不要。每个账号独占一个静态住宅 IP。
